开启调试模式
-代码修改后服务器自动重启

Windows: set FLASK_ENV=development
linux: export FLASK_ENV=development

使⽤app.run()⽅式可以启动flask应⽤，但是每次修改代码 之后，需要重新启动，这样对于调试就很不太⽅便。
启⽤调试模式，服务器会在代码修改后⾃动重新载⼊，并在发⽣错误时提供⼀个能获得 错误上下⽂及可执⾏代码的调试⻚⾯
开启调试模式会成为⼀个巨⼤的安全隐患，因此绝不能在⽣产环境中开启调试。
启⽤flask的调试模式有两种⽅法：

  1. 直接设置app的debug为true: app.debug=true
  2. 把debug=true作为参数，传⼊到run⽅法：app.run(debug=true)

上下文对象
上下文就是工作的环境
每一段程序都有很多外部变量。只有像Add这种简单的函数才是没有外部变量的。一旦你的一段程序有了外部变量，这段程序就不完整，不能独立运行。你为了使他们运行，就要给所有的外部变量一个一个写一些值进去。这些值的集合就叫上下文。

-应用上下文
-请求上下文
-在分派请求之前激活应用上下文和请求上下文
请求分派：就是分派 url对应视图

-在请求处理完成后将其删除
整个请求的处理过程，都会在这个上下⽂对象中进⾏

应用上下文 是⼀个上下⽂对象，⽽不是应⽤，
应用指的是flask 的app是确定请求所在的应⽤
current app当前应用的实例
g处理请求时的临时存储对象,每次请求都会重设这个变量

请求上下文对象
request 请求对象,封装了客户端发出的HTTP请求中的内容
session 用户会话(dict),各请求之间的数据共享

请求报文指的是请求上下文对象中的request
请求报文常用参数

  • method:请求的类型(GET/POST/OPTIONS等)
  • form:POST请求数据dict
  • args:GET请求数据dict
  • values:POST请求和GET请求数据集合dict
  • files:上传的文件数据dict
  • cookies:请求中的 cookie dict
  • headers:HTTP请求头

Flask请求报⽂的理解，Flask⽀持四种请求钩⼦，
思考:如下场景如何实现?
-每个请求中都要验证用户信息(是否已登录、是否有权限访问）
限制某些IP的恶意请求
使用钩子函数可以减少重复代码的编写,便于维护
request就是flask中代表当前请求的request对象
当我们需要对请求进⾏预处理和后处理时，就可以⽤Flask提供的钩⼦函数，⽤他们来注 册在请求处理的不同阶段执⾏的处理函数

  • before_first_request 服务器初始化后在处理第一个请求到达前执行
  • before_request 每一个请求到达前执行
  • after_request 如果没有未处理的异常抛出，在每次请求处理完成后执行,如果请求过程中产生了异常,则不执行
  • teardown_request 每次请求处理完成之后执行,如果请求过程中产生了异常也执行

模板引擎的作用
一是使用户界面（HTML）与业务数据（后端逻辑代码）分离。
二是可以将写好的HTML模板代码渲染成纯HTML页面，就可以实现动态数据展示了。
模板引擎的实现方式有很多，最简单的是“置换型”模板引擎，这类模板引擎只是将指定模板内容（字符串）中的特定标记（子字符串）替换一下便生成了最终需要的业务数据（比如网页）。置换型模板引擎实现简单，但其会多消耗服务器一些性能，而且效率低下，无法满足高负载的应用需求（比如有海量访问的网站），因此还出现了“解释型”模板引擎和“编译型”模板引擎等。

模板其实是⼀个包含响应⽂本的⽂件，其中⽤变量表示动态部分，
告诉模板引擎其具体的值需要从使⽤的数据中获取。
