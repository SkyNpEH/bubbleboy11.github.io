协程：协同多任务

  • 协程是轻量级线程，单线程，分解一个线程成为多个 “微线程”。
  • 可以执行并发任务，原因是协程把切换的权利交给程序员调度而不是操作系统，与程序员决定在哪些环节进行切换。协程可以处理上万的并发，多线程即不可以，因为切换成本太大，会耗尽计算机资源
  • 不需要锁机制，因为是单线程
  • 对多核CPU的利用——多进程+协程
    协程是为非抢占式多任务产生子程序的计算机程序组件，协程允许不同入口点在不同位置暂停或开始执行程序。从技术的角度来说，“协程就是你可以暂停执行的函数”

基于生成器的协程 Generator Based Coroutine

python3.5之前没有原生协程，是基于生成器（yield）来实现的协程。

  • pep 342(Coroutines via Enhanced Generator)增强生成器功能
  • 生成器可以通过yield暂停执行和产出数据
  • 同时支持send()向生成器发送数据和throw()向生成器抛异常
  • 协程需要使用 send(None) 或者 next(coroutine) 来预激 (prime)才能启动
  • 在 yield处协程会暂定执行
  • 单独yield value 会产出值给调用方
  • 可以单独coroutine.send(value) 来给协程发送值，发送的值会赋值给 yield表达式左边的变量
  • 协程执行完后（没有遇到下一个yield语句）会抛出StopIteratio异常

调用next()，函数将不断的执行语句直到遇到下一个yield的位置，yield在函数语句执行暂停，直到下一次遇到又会暂停

日志记录配置
logging是一个python标准库
logging的四个部分（配置logging中很重要）：
• logger——日志记录入口
logger的级别：
○ DEBUG：用于调试目的的底层系统信息
○ INFO: 普通的系统信息
○ WARNING： 警告，较小的问题，不影响执行顺序
○ ERROR: 错误，较大的问题
○ CRITICAL: 严重，致命的问题
logger记录内容：
○ django 记录所有内容
○ django.request 记录？（官方文档说是记录请求，但是使用后发现没有写入任何内容。。。
○ django.server 记录请求
○ django.template 记录与渲染模板相关的内容
○ django.db.backends 记录数据库操作
○ django.security.* 记录与安全相关的内容
• Handler——决定处理logger中消息的方式
比如写入文件‘log_file’，控制台输出’console‘，发送邮件’mail_admins’等等
• Filters——对日志进行条件控制
• Formatters——日志记录的文本顺序（格式）
在view视图中写入logging的方法
确保切换到了debug模式，否则本地运行无法读取静态文件，服务器端运行则需要部署nginx。

查询条件的使用
结果集QuerySet常用方法
Queryset表示从数据库中取出来的对象的集合
它可以含有零个、一个或者多个过滤器（filter）（相当于数据库的where）
从模型的Manager那里取得QuerySet
QuerySet的筛选结果本身还是QuerySet
QuerySet是惰性的（不是每一次都去数据库执行sql，比如：把用户取出一个结果集，查找一次后不会去数据库中查找了，会在用户结果集中查找）

get:返回一条且只能有一条数据，返回值是一个对象，参数可以写查询条件。

注意：对一个QuerySet实例对象，可以继续调用上面所有函数

  • get_or_create()——有则返回，没有则创建记录，可以实现链式的调用，通过第一次的过滤还可以再过滤
  • count()——返回记录行数之和 返回结果集中数据的数目，返回值是一个数字
  • none()——创建一个空的结果集
  • using()——使用指定的数据库查询(在settings中添加多数据库支持)

队列 是有序数据集合

**队列和栈一样，队列是一种操作受限制的线性表。
进行删除操作的端称为队头，称为前端(front)，
进行插入操作的端称为队尾，称为后端(rear)。

数据项总是开始排在队伍的后端，慢慢向前走，直到排到最前面，轮到它的时候离开队列。
刚进来的数据排在后端，待在队伍里时间最长的在前端，
这种排列规则叫做FIFO(first-in first-out)，“先进先出”

Python 中使用 collections 中的函数 deque() double-ended queue
底层是链表，因此头部和尾部是等价的，其插入和删除操作在表的两端进行，都是 O(1) 时间复杂度；
删除队列头部的数据项，不需要参数，返回值是被删除的数据，
队列本身有变化，直接把head指针移动到下一个元素，再把值弹出即可。
是一种具有队列和栈的性质的数据结构。
但是对于列表可以很方便向后面去追加元素，但如果想在第一个位置插入元素就需要移动后面所有的元素，因此在这里效率不如 collections.deque() 。
deque对于list的性能优势，
主要是在两端反复添加大量元素时差异才会比较明显，
而对于构建一个全新的集合，list反而比deque还要快一些。

栈（stack）又名堆栈，它是一种运算受限的线性表。

其限制是仅允许在栈的栈顶进行插入和删除运算。
这一端被称为栈顶，相对地，把另一端称为栈底。
用 append() 方法把新元素放到栈顶元素的上面，使之成为新的栈顶元素，称作是进栈，入栈或者压栈

从一个栈删除元素又称作出栈或退栈，用不指定索引的 pop() 方法可以把一个元素从堆栈顶释放删除，使其相邻的元素成为新的栈顶元素。

栈可以用顺序表实现，也可以用链表实现。
列表方法使得列表可以很方便的作为一个堆栈来使用

最先进入的元素最后一个被释放
先进后出 后进先出

栈的本质是递归。所以可以借助栈理解递归；
栈顶是递归终止条件，栈的每一层是递归的方法体

取值列表[-1]，因为它只能从栈顶来取值，相当于取列表的最后一个值，所以用索引-1.

SDK API

发表于 2020-08-20 更新于 2021-04-27

软件开发工具包 SDK Software Development Kit

辅助开发某一类软件的相关文档说明书、范例和工具的集合都可以叫做SDK。

SDK被开发出来是为了减少程序员工作量，越容易开发越好

比如——

有公司开发出某种软件的某一功能，把它封装成SDK（比如数据分析SDK就是能够实现数据分析功能的SDK），出售给其他公司做开发用，其他公司如果想要给软件开发出某种功能，但又不想从头开始搞开发，直接付钱省事。

SDK想象成一个虚拟的程序包，在这个程序包中有一份做好的软件功能，这份程序包几乎是全封闭的，只有一个小小接口可以联通外界，这个接口就是API。

这个SDK提供了这么多API
SDK 是可以下载的。API 是写在文档里的

你是个男的
你要开发一个BABY
SDK
就是房间、床、你老婆等一系列相关的工具、文档
API
就是你老婆开放的接口

有一杯密封饮料，它的名字叫做“SDK”。
饮料上插着吸管，吸管的名字叫“API”。

把你叫做“XX系统”。
如果你想喝到SDK里的饮料（让系统拥有SDK中的功能），你必须通过API这根吸管来实现（通过API连接你的系统和SDK工具包），否则你就喝不到饮料。

API Application Programming Interface 应用程序编程接口

是一些预先定义的函数，
目的是提供应用程序与开发人员基于某软件或硬件得以访问一组例程的能力，
而又无需访问源码，或理解内部工作机制的细节。

  • 预先定义好的程序代码

  • 屏蔽实现的细节

  • 被调用来获取信息或实现某些功能

  • 我有一些功能

  • 你们呢不用关心我是怎么实现的

  • 我给你一个访问地址

  • 你通过地址来找我，按照我规定的格式提供我需要的信息

  • 我会给你你需要的信息反馈

连接前端后端及移动端，系统与系统、模块与模块、服务与服务直接相互调用的入口，
开发者分工协作封装成接口，其他开发者只要按照对应格式，发送一些必要参数，就能使用功能

![description](./SDK API/接口.png)

常见接口类型

  • HTTP接口 传输文本表单数据、json类型的对象、xml类似的数据
  • RPC 远程方法调用，使用分布式系统，调用部署到其他服务器，远程调用希望像本地调用一样方便，让调用者感知不到远程调用的逻辑，不需要搞一串发起http请求的代码
  • SOAP - Simple Object Access Protocol 简单面向对象访问协议，一种协议规范，是一种轻量的，简单的，基于HTTP协议，使用XML作为默认的存储格式。

SOAP与REST区别：

  • 安全性：SOAP会好于REST

  • 效率和易用性：REST更胜一筹

  • 成熟度：总的来说SOAP在成熟度上优于REST

  • Web Servicez 是一种跨编程语言 和 跨操作系统平台的远程调用技术。基于SOAP协议的PRC实践方案，可以拿到远程对象并调用其属性和方法，

  • REST API
