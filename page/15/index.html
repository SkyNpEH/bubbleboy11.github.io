前后端分离

发表于 2021-04-02 更新于 2021-07-26 分类于 web

前端：客户端：浏览器、手机平板

在传统的Web应用开发中，将浏览器作为前后端的分界线。
将浏览器中为用户进行页面展示的部分称之为前端，
而将运行在服务器为前端提供业务逻辑和数据准备的所有代码统称为后端。

开发流程

  • 需求->UI->前端/后端->联调->测试->上线

前端开发

  • 分析、拆分、封装、模块化、组件化、组件通信

后端开发

  • 分析、模块化、ORM设计、接口标准

前端后端的两种协作方式，一种可以称之为是服务器端渲染，一种是叫做前后端分离。
它们的区别
HTML 是在哪儿生成的，浏览器和服务端传递的是什么。
在前后端分离的方式，浏览器和服务端传递的是内容，指网页是在浏览器端生成的，
在服务端渲染的过程中，传递的是 HTML 网页。

服务器渲染

指网页直接在服务端生成，浏览器拿到的是一整个网页，CSS 和 JS 的部分是在浏览器端执行的，网页的内容也是由服务器端生成的。
传统的web开发框架里面都是使用MVC模式，
后端处理业务逻辑并带上数据返回模版页面进行渲染，渲染完HTML之后再去返回给浏览器让浏览器去展示。
django是后端路由返回模板，使用Django自带的Template模版引擎，这种通常需要自己利用HTML+CSS+Jquery的方式总感觉是上一个时代的做法，
新建package为templates，通过from django.shortcuts import render， 视图返回render

java的springboot也有自己的模版引擎，有不同的模版渲染语法。
前端写好的页面还需要后台程序员转换成jsp或模版文件才能使用。

前端和后端开发人员经常会同时去修改模板文件，这样可能会导致经常会有冲突，而且前端和后端基本上同时在一个项目仓库，然后基本上同时去部署。
项目比较大的时候就会显得代码仓库比较臃肿也比较乱。

开发流程
提出需求 –> 前端页面开发 –> 翻译成模板 –> 前后端对接 –> 集成遇到问题 –> 前端返工 –> 后端返工 –> 二次集成 –> 集成成功 –> 交付上线

前后端分离：通过HTTP连接前后端

约定数据交互接口规范，并行的进行开发和测试
前端只需要关注页面的开发，并通过ajax方法发HTTP请求用后端的接口（API）获取后端提供数据，将数据通过原生js渲染出来到页面展示或增加修改，或者用vue，react等框架框架自带路由跳转控制和模版页面渲染，
通信数据一般采用json数据进行交互，
同时前端的提交数据库也能有对应的操作，所以前后端至此是打通了。
mvc中的c 交给前端

后端，服务器，接受HTTP请求，只负责按照约定的数据格式为前端提供的可调用的数据接口api和业务逻辑，不负责将数据渲染到模板页面上，不会再用到模版渲染语言，Django的Template的DTL是用不上的，setting.py的TEMPLATES配置也不用了

后端写服务使用Django rest framework或者fastapi来快速的提供API，通过from django.http import JsonResponse，视图返回return JsonResponse({'sn': obj.sn}, status=201)

API 在web领域就是url path

提出需求 –> 约定接口规范数据格式 –> 前后端并行同步开发 –> 前后端对接 –> 前端调试效果 –> 集成成功 –> 交付上线

用nginx做前端服务器，通过nginx转发请求到后端。
nginx本身就是http服务器，返回给前端没有数据

使用前后端分离开发好处

耦合（Coupling）：软件之间的依赖性、一个软件结构内不同模块之间互连程度的度量
内聚（Cohesion）：同一模块各元素之间的紧密程度

  1. 提升开发效率。后台开发语言和模板高度耦合，导致开发语言依赖严重，后端语言的切换成本很高。前后端分离以后，可以实现前后端代码的解耦，只要前后端沟通约定好应用所需接口以及接口参数，便可以开始并行开发，无需前后端互相等待对方的开发工作结束。在这种情况下，前后端工程师都可以只专注于自己的开发工作，有助于打造出更好的团队。
    即使需求发生变更，只要接口与数据格式不变，后端开发人员就不需要修改代码，只要前端进行变动即可将页面渲染的任务交给了客户端并大大降低了服务器的压力，可以让服务器专注在数据处理，

  2. 增强代码的可维护性。前后端分离后，应用的代码不再是前后端混合，只有在运行期才会有调用依赖关系，维护成本降低，再不会牵一发而动全身。当你的代码变得简明且整洁时，代码的可读性和可维护性都会有质的提升。更有利于调试(mock)、测试和运维 不同服务器上部署，前后端只要能通信就可以.精准的单元测试，精准的 bug 定位和修复

  3. 支持PC、APP、PAD多终端和服务化架构。
    接口复用（公用），减少开发量，同一套数据接口可以为不同的终端提供服务，更有助于打造多终端应用；4. 由于后端提供的接口之间可以通过HTTP(S)进行调用，有助于打造服务化架构（包括微服务）。

  4. SPA(Single Page web Application)单页面开发模式是流行趋势。

前后端分离缺点

  1. 前后端学习门槛增加
  2. 数据依赖导致文档重要性增加
  3. 前端工作量加大
  4. SEO(Search Engine Optimization)（搜索引擎优化）的难度增大
  5. 后端开发模式迁移增加成本
  6. 跨域问题

不同维度比较
传输量：前后端分离，传输量会减少；服务端渲染，传输量大，内容会有大量冗余。
渲染：前后端分离，多了渲染数据的过程；服务端渲染，首屏渲染性能低。
展示：前后端分离，数据展示由前端处理，后端只提供内容；服务端渲染，传输的是 HTML。
动作：前后端分离，页面交互是由前端来决定；服务端渲染，大部分是后端，极少部分由前端完成。
SEO：前后端分离，架构搭建采取 SPA 方式，不利于 SEO；服务器端渲染，生成的是网页，所以对 SEO 支持友好
