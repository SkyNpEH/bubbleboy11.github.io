  1. git是维护三棵树来实现版本控制的
    1）Working(Directory):存放工作目录的地方 添加、编辑、修改文件等操作。
    2）Stage(Index):暂存修改是一个个的文件，即将存放到仓库
    3）Repository(HEAD):终确定的文件保存到Git仓库成为一个新版本。最终存放版本数据的位置，HEAD是最新版本的指向数据

GitHub
(1) 仓库 (Repository)
仓库是用来存放项目代码，每一项目对应一个仓库，多个仓库则会有多个仓库。

(2) 收藏 (Star) stars:>1000 # stars数量大于1000的项目
收藏别人的仓库，方便自己查找

(3) 复制/克隆项目 (Fork) forks:>4 # stars数量大于4的项目
别人仓库的代码可以克隆到自己的账号下的中，可以对仓库进行更改。自己在仓库中对bug进行更改不会影响原作者的仓库，但可以通过向原作者发起请求 (Pull Request)。

(4) 关注(Watch)

(5) 事物卡片(Issue)
发现别人的仓库代码出现Bug或有疑问时，可以通过Issue来和代码的作者进行咨询和讨论。然后作者可以收到别人的提问，然后可以回复。回复完成可以关闭当前的Issue。

✨✨版本的前进和后退需要通过 指针 HEAD 来进行控制。
注：命令参数有 soft, mixed, hard ** 三种类型：**
(1) 可省略的默认参数 mixed 表示本地Git仓库移动HEAD指针，重置暂存区。
git reset (–mixed) HEAD~ 移动HEAD的指向上一个快照
git reset HEAD~3,代表回滚最近3次的提交。
执行后 再用git log也看不到回滚的提交记录了
git reset 版本快照ID // ‘bc124d4a1ca’是git log 后显示的黄色提交序号,
不需要写全,但是要保证写的快照ID是唯一，回滚到指定位置,之前的提交会全部回滚到暂存区域

(2)soft表示本地Git仓库移动HEAD指针。
git reset –soft HEAD~ 移动HEAD的指向,将其指向上一个快照

(3) 参数 hard 本地Git仓库移动HEAD指针，重置暂存区，重置工作区。
git reset HEAD //不写将回滚最后一次提交的所有文件，如果指定了将只回滚选中的文件

HEAD控制的方式有三种：
(1) 使用版本号进行操作(常用)，可以前进可以后退：
命令格式为：git reset –hard 版本id
(2) 使用^符号进行操作，只能后退：
命令： git reset –hard HEAD^ 表示后退一个版本
命令： git reset –hard HEAD^^ 表示后退两个版本

(3) 使用符号进行操作，只能后退：
git reset –hard HEAD 表示后退一个版本（快照），HEAD的指向,将其指向上一个
将HEAD移动后指向的快照回滚到暂存区域，将暂存区域的文件还原到工作目录
git reset –hard HEAD~~ 表示后退两个版本；
git reset –hard HEAD5 表示后退5个版本；
git reset –hard HEADn 表示后退n个版本；
git reset –hard 分支名 切换HEAD指向的分支以及HEAD本身，原本master的仓库消失了

4.diff 对比找不同
git diff 文件名 比较暂存区域与工作目录不同的地方，也就是所更改的地方
git diff HEAD 文件名 //工作目录和最新提交的Git仓库文件比较
git diff 版本号 文件名 工作区和Git仓库 的不同版本文件比较，
git diff 快照1ID 快照2ID
git diff 不添加文件名进行比较时，将自动比较多个相同文件名文件，比较两个历史快照

  2. git diff HEAD //比较当前工作目录和最新提交的快照

4比较暂存目录和Git仓库中的快照(git add后的)

合并分支的时候可能会产生冲突，下面举一个例子进行说明：
上一个操作合并分支，则master分支下的文件和feature_sum分支下的文件相同。
假设用户A对master分支下 test_sum.cpp 文件进行修改然后提交到仓库，用户B对feature_sum分支test_sum.cpp文件进行修改然后提交到仓库。用户A和用户B对同一个文件进行修改，那么修改的文件内容在合并时会产生冲突，系统处于合并状态，系统不知道该如何操作，则会出现报错，此时就需要人工

Python的多进程

标准库有两个： Multiprocessing 和 Concurrent.futures

利用multiprocess模块的Pool类创建进程池

很多时候系统都需要创建多个进程以提高CPU的利用率，当数量较少时，可以手动生成一个个Process实例。
当进程数量很多时，或许可以利用循环，但是这需要程序员手动管理系统中并发进程的数量，有时会很麻烦。
这时进程池Pool就可以发挥其功效了。
Pool类可以提供指定数量的进程供用户调用，当有新的请求提交到Pool中时，如果进程池还没有满，就会创建一个新的进程来执行请求。如果池满，请求就会告知先等待，直到池中有进程结束，才会创建新的进程来执行这些请求。
创建进程池：先写出创建进程池的方法，之后往进程池中放入进程即可

二、使用进程池的优势
不仅能制定进程创建的数量，还可以避免进程的创建与关闭带来的消耗。
进程池创建一次之后不会关闭，可以重复使用，进程池关闭的时候，内部的进程也会关闭

concurrent.futures 模块

Python3.2带来了concurrent.futures 模块，这个模块包含了线程池和进程池、管理并行编程任务、处理非确定性的执行流程、进程/线程同步等功能。
对 threading 和 multiprocessing 进行了高级别的抽象, concurrent.futures 底层还是用的 threading 和 multiprocessing 这两个模块, 相当于在这上面又封装了一层, 所以速度上会慢一点, 这个是架构和接口实现上的取舍造成的。
暴露出统一的接口, 方便开发者使用并发和并行的代码更简单了

该模块提供以下对象和函数：
可以利用multiprocessing实现真正的平行计算。
核心原理是：concurrent.futures会以子进程的形式，平行的运行多个python解释器，从而令python程序可以利用多核CPU来提升执行速度。由于子进程与主解释器相分离，所以他们的全局解释器锁也是相互独立的。每个子进程都能够完整的使用一个CPU内核。

关于这部分的内容推荐大家阅读《Python并行编程》。

这些接口由抽象类 Executor 定义。
这个模块提供了两大类型，
一个是执行器类 Executor，另一个是 Future 类。

期程对象：concurrent.futures.Future
模块函数：concurrent.futures.wait
执行器对象：concurrent.futures.｛Executor,ThreadPoolExecutor,ProcessPoolExecutor}

ProcessPoolExecutor 是 Executor 的子类，它使用进程池来实现异步执行调，使用 multiprocessing 回避 Global Interpreter Lock 但也意味着，作为进程的函数只可以处理和返回可序列化的对象， __main__ 模块必须可以被子进程导入，这意味着 ProcessPoolExecutor 不可以工作在交互式解释器中。

Python多进程可通过派生（fork或spawn）新的进程来执行其他任务

fork分支创建：会直接复制一份自己给子进程运行，并把自己所有资源的handle 都让子进程继承，因而初始化创建速度很快，但更占用内存资源。只支持有fork函数的Unix/Linux系统，不支持Windows，

spawn分产创建：只会把必要的资源的handle 交给子进程，因此创建速度稍慢。
只继承必要的资源，所以更省内存，高性能计算需要让程序运行很久，因此为了节省内存以及进程安全

  • PyPy，基于Python语言开发的解释器。 PyPy是另一个Python解释器，它的目标是执行速度。PyPy采用JIT技术，对Python代码进行动态编译（注意不是解释），所以可以显著提高Python代码的执行速度。
    绝大部分Python代码都可以在PyPy下运行，但是PyPy和CPython有一些是不同的，这就导致相同的Python代码在两种解释器下执行可能会有不同的结果。如果你的代码要放到PyPy下执行，就需要了解PyPy和CPython的不同点。

  • Jython，运行在Java平台的解释器，直接把Python代码编译成Java字节码执行。

  • IronPython，只运行在微软.Net平台上的Python解释器，可以直接把Python代码编译成.Net的字节码。

如果要和Java或.Net平台交互，最好的办法不是用Jython或IronPython，
而是通过网络调用来交互，确保各程序之间的独立性。

在其他语言当中，一个进程里面开启多个线程，每个线程都可以给一个cpu去使用，
比如我现在开启了一个进程，这个进程当中含有几个线程，如果我现在有多个cpu，每一个线程是可以对应相应的CPU的。在多CPU系统中，为了最大限度的利用多核，可以开启多个线程。

GIL Global Interpreter Lock (全局解释器锁)

  • 这个GIL并不是python的特性，他是在Cpython解释器里引入的一个概念，而在其他的语言编写的解释器里就没有这个GIL
  • CPython GIL 就是 Cpython解释器的全局解释锁进程内有一条主线程，以及多条用户程序的执行线程。
  • 多个线程不能同时并行执行，即使在多核CPU平台也不能利用 多个CPU核心执行 的优势，只在单核CPU系统中工作，不存在真正的并发

全局锁是主要的作用影响

  • 因为多线程的编程方式，使得线程之间数据的一致性和状态同步难以把控，为了解决数据不能同步的问题，设计了GIL全局解释器锁。

  • 由于官方的Cpython解释器的内存管理并不是线程安全的（如果有多个线程话它可能会同时去修改同一个对象，这样可能会出现一些问题），因此为了支持多线程编程。

  • 保护多线程情况下对Python对象的访问

  • Python 官方解释器 的每个线程要获得执行权限

  • 有了GIL，保证在同一时刻，一个解释器进程下只有一个线程在运行、执行字节码。
    Cpython使用简单的锁机制避免多线程同时执行字节码

  • 在Cpython解释器中，同一进程中假如有多个线程运行，当python代码有一个最高优先级的线程开始访问解释器的时候，一个线程在运行python程序的时候会霸占python解释器（加了一把锁），GIL会把这个线程给锁上，
    此时此刻使该进程内的其他线程无法运行，无法对解释器的资源进行访问，需要等这个线程分配的时间到了、该线程运行完后这个线程把GIL释放掉，其他线程才能运行。
    如果线程运行过程中遇到耗时操作，则解释器锁解开，使其他线程运行。
    所以在多线程中，线程的运行只能由操作系统负责线程优先级的排序，并不是同时进行，用户无法控制线程的选择

这类似于给线程加锁 threading.Lock().acquire()，解锁 threading.Lock().release() 一样。

  • 单个进程内的并发任务：如果你需要在单个进程内继续使用并发的特性，可以考虑使用threading模块，不能利用多核CPU，
    如果想要充分地使用多核CPU的资源，在python中大部分情况需要使用多进程multiprocessing模块。

首先区分CPU和IO密集程序

  • 一般程序分为CPU密集还有IO密集
    • CPU密集：把大量时间花在计算上面， 对视频进行高清解码，难以利用多核优势，任务越多， 花在任务切换的时间就越多， CPU执行任务的效率就越低，
    • IO密集：把大量时间花费在了等待上面。(比如网络传输、文件操作，磁盘硬盘读写)
      在等待时间中CPU是不需要工作的，那你在此期间提供双CPU资源也是利用不上的，那么为什么多线程会对IO密集型代码有用呢？这时因为python碰到等待会释放GIL供新的线程使用，实现了线程间的切换。
      IO期间会释放GIL，对IO密集程序影响不大
      解释图：这个图里面有三个线程Thread1，Thread2，Thread3，在处理网络请求的过程中，虽然一个时间只能执行一个线程，但是在等待网络请求的时候来释放全局解释锁切换到不同线程Run运行，这就是对IO密集的程序会小一些。

如何规避GIL影响

  • CPU密集可以使用多进程+进程池来充分利用多核，2个CPU干活肯定比一个CPU快很多。
    用cython编写一些扩展来转换成cython代码。用C语言编写。
  • （cython不是Cpython解释器，它是一个用来把python程序转换成C代码一个扩展）
  • IO密集型任务：使用多线程 / 协程。原因是一个线程在IO时被阻塞，会被切换到另一个线程，线程切换后，I/O 操作仍然在进行，线程 1 在进行 I/O 操作时，线程 2 可以获得 CPU 资源进行计算，虽然增加了切换成本，却提高了效率。
  • 通过pypy解释器来执行python脚本不含有GIL

GIL实现

解释图：每隔一段时间我们尝试去释放当前线程的锁，然后让其它线程尝试去获取锁并且去执行

GIL执行顺序

  1. 设置GIL
  2. 切换进一个线程去运行
  3. 执行下面操作之一：
  • 指定数量的字节码指令
  • 线程主动让出控制权（可以调用time.sleep(0)来完成）
  1. 把线程设置回睡眠状态（切换出线程）
  2. 解锁GIL
  3. 重复上述步骤
